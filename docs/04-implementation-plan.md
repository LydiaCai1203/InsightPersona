# 04 - 实施计划

---

## 一、前置依赖（Blockers）

开始实施前，必须确认以下信息：

| 项目 | 状态 | 说明 |
|------|------|------|
| 数据文件 | 待提供 | CSV / JSON / Parquet 均可 |
| `user_id` 字段 | 待确认 | 用户唯一标识，User 层聚合的前提 |
| `task_id` / `session_id` 字段 | 待确认 | 任务/会话标识，Task 层聚合的前提 |
| `timestamp` 字段 | 待确认 | 时间戳，时间维度特征的前提 |
| `role` 字段 | 待确认 | 区分 user / assistant 消息 |
| 消息内容字段名 | 待确认 | 用于规则提取和 LLM 标注 |
| 平均 turns/task | 待确认 | 影响 LLM 标注的 token 用量估算 |
| 数据脱敏状态 | 待确认 | 是否可以直接处理原始内容 |

## 二、实施步骤

### Phase 0：数据探查（0.5 天）

```
输入：原始数据文件
输出：数据概况报告（字段、分布、缺失率、数据量确认）
```

1. 加载数据，确认 schema
2. 统计：总 turn 数、总 task 数、总 user 数、平均 turns/task
3. 检查时间戳分布：时间范围、活跃趋势
4. 抽样检查消息内容质量

### Phase 1：Turn 层规则提取（0.5 天）

```
输入：原始对话数据
输出：每条 turn 的结构化特征（见 01-feature-architecture.md 1.1 节）
成本：零
依赖：无
```

1. 对每条消息运行正则/规则提取
2. 生成 `turn_features` 表
3. 按 `task_id` 聚合生成 `task_turn_agg` 表（见 01-feature-architecture.md 1.3 节）

### Phase 2：Task 层 LLM 标注（2-3 天）

```
输入：每个 task 的完整对话文本 + task_turn_agg
输出：每个 task 的 19 个标注字段
成本：~$15-22
依赖：Phase 1（task_turn_agg 作为 LLM prompt 的补充上下文）
```

#### 2a：Prompt 设计与验证（0.5 天）

1. 从数据中随机抽取 50 个 task
2. 设计 Batch 1 prompt（9 个客观字段）
3. 人工核对 50 个样本的标注结果
4. 迭代优化 prompt 直到准确率 > 85%
5. 设计 Batch 2 prompt（10 个推断字段）
6. 人工核对，接受 60-80% 准确率

#### 2b：批量标注（1-2 天）

1. Batch 1：对全部 ~100K task 标注 9 个客观字段
2. 质量检查：抽检 100 个样本
3. Batch 2：对全部 ~100K task 标注 10 个推断字段
4. 质量检查

#### Batch 分批策略

| Batch | 字段 | 准确率预期 | Prompt 策略 |
|-------|------|-----------|------------|
| Batch 1（客观事实） | task_type, tech_stack, domain, task_complexity, role, industry, task_outcome, is_within_capability, deliverable | > 85% | 强调"只基于证据"，要求引用对话原文 |
| Batch 2（推断性） | user_skill_signal, intent, scope, ai_dependency, project_phase, comfort_zone, time_pressure, team_signal, codebase_scale, decision_authority | 60-80% | 允许 unclear，提供推断指引和示例 |

### Phase 3：User 层聚合（0.5 天）

```
输入：task_features 表 + task_turn_agg 表
输出：user_features 表（分布型、主标签、行为统计、生命周期特征）
成本：零
依赖：Phase 2
```

1. 按 `user_id` 聚合所有 task 层标签，生成分布型特征
2. 从分布中提取主标签
3. 计算行为统计特征
4. 计算生命周期特征

### Phase 4：时间维度特征（0.5 天）

```
输入：user_features 表 + 时间戳数据 + task_features 表
输出：时间维度特征（节奏、演化、生命周期阶段、间隔模式、关键时刻）
成本：零
依赖：Phase 2（演化类特征需要 task 层标签）
```

注意：节奏类和间隔类特征只依赖时间戳，可以与 Phase 2 并行。演化类特征必须等 Phase 2 完成。

### Phase 5：分析路线执行（3-5 天）

```
输入：完整的 user_features 表（含时间维度）
输出：11 条分析路线的全部产出物
成本：零（纯统计/ML）
依赖：Phase 3 + Phase 4
```

| 路线 | 预计耗时 | 方法 |
|------|---------|------|
| 路线一：用户分群 | 1 天 | UMAP + HDBSCAN |
| 路线二：激活留存 | 0.5 天 | Cohort 留存 + 逻辑回归 |
| 路线三：商业化分层 | 0.5 天 | 评分模型 + 分层 |
| 路线四：产品能力 Gap | 0.5 天 | 二维矩阵 |
| 路线五：用户演化路径 | 0.5 天 | 状态转移矩阵 + Sankey 图 |
| 路线六：跨维度交叉 | 0.5 天 | 交叉表 + 统计检验 |
| 路线七：垂直行业深挖 | 0.5 天 | 频次统计 + 关联规则 |
| 路线八：交付物分析 | 0.25 天 | 分布统计 + 交叉 |
| 路线九：探索行为分析 | 0.25 天 | 分布统计 + 交叉 |
| 路线十：决策链分析 | 0.25 天 | 交叉矩阵 |
| 路线十一：项目阶段覆盖 | 0.25 天 | 分布统计 + 三维交叉 |

## 三、成本汇总

| 项目 | 成本 | 说明 |
|------|------|------|
| Turn 层规则提取 | $0 | 纯 regex/规则 |
| Task 层 LLM 标注 Batch 1 | ~$8-12 | GPT-4o-mini / Claude Haiku，~100K tasks x ~3K tokens |
| Task 层 LLM 标注 Batch 2 | ~$7-10 | 同上 |
| User 层聚合 | $0 | SQL/Pandas |
| 时间维度特征 | $0 | SQL/Pandas |
| 分析路线执行 | $0 | 统计/ML |
| **总计** | **~$15-22** | |

### 成本优化选项

- 如果预算紧张，可以先只跑 Batch 1（$8-12），用 9 个客观字段做路线 1-4 和路线 7
- Batch 2 的 10 个推断字段在路线 5、6、9、10、11 中才真正需要
- 可以先用 Batch 1 结果验证方案可行性，再决定是否跑 Batch 2

## 四、时间线

```
Day 0:     拿到数据
Day 0.5:   数据探查完成
Day 1:     Turn 层规则提取完成
Day 1.5:   LLM Prompt 设计与验证完成
Day 3:     Batch 1 标注完成
Day 3.5:   Batch 2 标注完成
Day 4:     User 层聚合 + 时间维度特征完成
Day 5-9:   11 条分析路线逐步执行
Day 9:     全部产出物交付
```

总耗时约 **7-9 个工作日**（从拿到数据开始算）。
